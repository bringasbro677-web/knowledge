QCASH Incident – “Lesson Learned Brings Comfort, Not Chaos”

Dalam penanganan insiden yang terjadi pada platform QCASH, tim Operation bersama konsultan melakukan serangkaian evaluasi untuk mengidentifikasi akar masalah serta merumuskan langkah-langkah mitigasi. Hasil evaluasi ini telah didokumentasikan sebagai bagian dari tacit knowledge, guna meningkatkan kapabilitas tim dan mencegah insiden serupa terulang di masa mendatang.

Fokus utama dari insiden ini berkaitan dengan pengelolaan database, performa sistem, serta penggunaan fitur teknologi yang sesuai dengan kebutuhan arsitektur layanan. Berikut poin-poin penting yang dapat menjadi referensi teknis ke depan:

1. PgBouncer

    PgBouncer digunakan sebagai connection pooler untuk membatasi jumlah koneksi langsung ke database. Hal ini berfungsi untuk menjaga kestabilan sistem serta mencegah kehabisan socket pada worker.

    PgBouncer tidak mendukung multithreading, performanya tetap dapat ditingkatkan dengan cara:

  • Menambah jumlah pod
  • Mengintegrasikan load balancer

 

2. Upgrade Versi Sistem

    Melakukan upgrade version harus dengan cermat, terutama jika sistem sudah stabil di production.

    Melakukan pengujian dengan meliputi:

  • Performa (Transaction Per Second)
  • Efisiensi penggunaan CPU

    Disarankan menggunakan versi yang lebih stabil (sub-version)

 

 

3. JSON tidak untuk query build

  • JSON digunakan untuk menyimpan data dinamis, seperti parameter transaksi.
  • JSON lebih tepat sebagai media penyimpanan yang fleksibel.
  • JSON tidak disarankan untuk keperluan query build karena query terhadap kolom JSON dapat menaikkan compute CPU.

     

4.  SCHEMA VS DATABASE

     Terdapat perbedaan mendasar terhadap makna Database pada PostgreSQL dan MySQL.

5.   Pelajaran dari Insiden

  • Keputusan untuk upgrade version terlalu jauh gapnya (misal version 1.12 lompat ke 1.22) perlu dipertimbangkan terkadang terdapat fitur yang sebenarnya kita tidak perlukan dan justru menambah kinerja computing CPU
  • Stress test dan benchmark sangat penting untuk mengetahui batas kemampuan Transaction per Second dari suatu version.
